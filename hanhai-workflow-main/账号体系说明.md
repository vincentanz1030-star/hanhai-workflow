# 账号体系说明

## 当前账号体系设计

### 1. 平级账号结构
目前系统采用**平级账号体系**，所有账号都是平等的，通过**角色（Role）**来区分权限，而不是账号层级。

### 2. 角色权限体系

#### 角色（Roles）
- **admin（管理员）**：系统管理员，拥有所有权限
- **operations（运营）**：运营人员，可管理项目、任务、销售目标、用户
- **product（产品）**：产品经理，可管理项目、任务、产品框架
- **copywriting（文案）**：文案人员，可管理项目、任务、协同
- **illustration（插画）**：插画师，可查看和编辑任务
- **detail（详情）**：详情页设计，可查看和编辑任务
- **purchasing（采购）**：采购人员，可查看和编辑任务
- **packaging（包装）**：包装设计，可查看和编辑任务
- **finance（财务）**：财务人员，可管理销售目标、协同
- **customer_service（客服）**：客服人员，可查看任务
- **warehouse（仓储）**：仓储人员，可查看和编辑任务

### 3. 品牌数据隔离
每个用户归属于一个品牌，只能查看和操作该品牌的数据：
- **he_zhe（禾哲）**
- **baobao（BAOBAO）**
- **ai_he（爱禾）**
- **bao_deng_yuan（宝登源）**
- **all（全部）**：仅管理员或特定角色可以使用

### 4. 多岗位支持
系统支持一人多岗，用户可以同时拥有多个角色：
```typescript
interface UserRole {
  role: string;        // 角色名称
  is_primary: boolean; // 是否主岗位
}
```

## 是否需要主账号/子账号区分？

### 当前方案：平级账号 + 角色权限
**优点**：
- ✅ 权限管理清晰，基于角色的访问控制（RBAC）
- ✅ 灵活性高，可以随意组合权限
- ✅ 符合现代权限管理最佳实践
- ✅ 易于扩展和维护

**缺点**：
- ❌ 没有账号层级关系
- ❌ 无法区分"主账号"和"子账号"
- ❌ 子账号无法被主账号管理

### 如果需要主账号/子账号体系

可以考虑以下方案：

#### 方案1：基于邀请的子账号系统
- **主账号**：可以创建和管理子账号
- **子账号**：由主账号创建，权限由主账号分配
- **特性**：
  - 主账号可以查看和删除子账号
  - 子账号不能登录到其他账号
  - 支持子账号数限制

#### 方案2：基于团队的账号体系
- **团队**：一个团队可以有多个成员
- **团队管理员**：拥有团队内管理权限
- **团队成员**：团队成员，权限由管理员分配

#### 数据库表设计（方案1示例）
```sql
-- 扩展users表
ALTER TABLE users ADD COLUMN parent_id UUID REFERENCES users(id);
ALTER TABLE users ADD COLUMN account_type VARCHAR(20) DEFAULT 'primary'; -- 'primary' 或 'sub'

-- 子账号关联表
CREATE TABLE sub_account_permissions (
  id UUID PRIMARY KEY,
  parent_id UUID REFERENCES users(id),
  sub_account_id UUID REFERENCES users(id),
  permissions JSONB, -- 自定义权限配置
  created_at TIMESTAMP DEFAULT NOW()
);
```

## 推荐方案

**推荐保持当前的平级账号体系**，原因：

1. **适合当前业务场景**：电商工作流程管理，各岗位协作即可
2. **权限已经足够清晰**：11个角色覆盖了所有业务场景
3. **维护简单**：不需要额外的账号层级管理
4. **符合最佳实践**：RBAC是业界标准的权限管理方案

### 如果确实需要主账号/子账号

建议实现**方案1（基于邀请的子账号系统）**：
1. 添加 `account_type` 字段区分主账号/子账号
2. 主账号可以邀请创建子账号
3. 子账号的权限由主账号分配
4. 添加子账号管理界面

## 使用示例

### 创建主账号
```bash
POST /api/auth/register
{
  "email": "admin@example.com",
  "password": "password123",
  "name": "管理员",
  "brand": "he_zhe",
  "role": "admin"
}
```

### 创建子账号（如果实现了子账号功能）
```bash
POST /api/users/invite-sub-account
{
  "email": "sub@example.com",
  "password": "password123",
  "name": "子账号",
  "permissions": {
    "project": ["view", "edit"],
    "task": ["view", "edit", "delete"]
  }
}
```

## 总结

当前系统已经具备完整的权限管理能力，通过角色和品牌两个维度实现权限控制。如果您的业务需要主账号/子账号的层级管理，可以基于现有架构进行扩展，添加账号类型字段和子账号管理功能。
